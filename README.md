# NavigationAnalyzer

NavigationAnalyzer is a CLI-first OSS tool for robot navigation debugging, benchmarking, visualization, evaluation, and observability.

The goal is a navigation failure analysis platform for ROS2/Nav2 and Autoware: Datadog/NewRelic-style observability for robotics, with reproducible evaluation and AI-readable diagnostics as first-class outputs.

## What Works Today

- Canonical JSON and ROS2 bag analysis.
- Nav2-oriented topic extraction for odometry, TF, cmd_vel, scan, pointcloud, path, costmap, goal, and recovery signals.
- Autoware-oriented topic extraction for localization, control commands, trajectories, mission goals, pointclouds, and vehicle state.
- Metrics:
  - success rate
  - path length
  - goal distance
  - time to goal
  - collision count
  - oscillation count
  - recovery count
  - path smoothness
  - minimum obstacle distance
  - final lateral, longitudinal, yaw, and stopped-duration metrics
  - Autoware route progress, straight-line route offset, and lanelet centerline metrics
- Rule-based failure taxonomy:
  - localization drift
  - oscillation
  - deadlock
  - narrow passage failure
  - dynamic obstacle freeze
  - planner divergence
- JSON and Markdown reports for AI agents and CI.
- Structured diagnostics for non-fatal evaluation warnings.
- React/Three.js/Plotly web visualization with trajectory replay, metrics, timeline, failure markers, and costmap/obstacle views.
- Docker workflow for Autoware planning simulator bags.

## Quick Start: No ROS Required

Create demo artifacts from the bundled JSON fixtures:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -e backend pytest
scripts/run_sample_demo.sh
```

Open these first:

```text
outputs/demo_sample/sample_failure/report.md
outputs/demo_sample/nav2_success/report.md
outputs/demo_sample/benchmark.md
```

The same command also writes machine-readable artifacts:

```text
outputs/demo_sample/sample_failure/analysis.json
outputs/demo_sample/nav2_success/analysis.json
outputs/demo_sample/benchmark.json
```

## Web UI

Serve an analysis artifact:

```bash
PYTHONPATH=backend python3 -m navigation_analyzer.cli.main serve \
  --analysis outputs/demo_sample/sample_failure/analysis.json \
  --port 8000
```

Start the frontend in another terminal:

```bash
cd frontend
npm install
npm run dev
```

Open the Vite URL, usually:

```text
http://127.0.0.1:5173
```

The UI can also open a local `analysis.json` directly with the file picker.

For multi-run comparison, open a local `benchmark.json` with the benchmark file picker. A quick demo benchmark is generated by:

```bash
scripts/run_sample_demo.sh
```

Then load:

```text
outputs/demo_sample/benchmark.json
```

Visual regression check:

```bash
cd frontend
npm run check:visualization
```

This captures desktop and mobile screenshots in `outputs/frontend-check/` and verifies that the Three.js canvas is not blank.

## CLI

Analyze one run:

```bash
navigation-analyzer analyze \
  --bag examples/sample_bag/sample_navigation.json \
  --config config/default.yaml \
  --out outputs/sample
```

Convert a ROS2 bag or canonical JSON to the canonical schema:

```bash
navigation-analyzer convert \
  --bag examples/sample_bag/sample_navigation.json \
  --config config/default.yaml \
  --out outputs/navigation_run.json
```

Compare multiple runs:

```bash
navigation-analyzer benchmark \
  --bag examples/nav2_sim_success_003/navigation_run.json \
  --bag examples/sample_bag/sample_navigation.json \
  --config config/default.yaml \
  --report outputs/benchmark.md \
  --out outputs/benchmark.json
```

The first `--bag` is the baseline for regression diffs in `benchmark.json` and `benchmark.md`.

Gate a benchmark with thresholds:

```bash
navigation-analyzer benchmark \
  --bag examples/nav2_sim_success_003/navigation_run.json \
  --config config/default.yaml \
  --thresholds config/benchmark_nav2.yaml \
  --fail-on-regression \
  --report outputs/nav2_gate.md \
  --out outputs/nav2_gate.json
```

Render a Markdown report from an existing artifact:

```bash
navigation-analyzer report \
  --analysis outputs/sample/analysis.json \
  --out outputs/sample/report.md
```

Analyze a unitree-nav-sim `record_nav_demo.py` smoke log directly:

```bash
PYTHONPATH=backend python3 -m navigation_analyzer.cli.main analyze \
  --bag /path/to/unitree-nav-sim/log/smoke_nav2_g1_full/isaac.log \
  --out outputs/unitree_t7_contact_checked
```

The log importer preserves goal distance, commanded velocity, minimum obstacle
clearance, PhysX obstacle overlap count, and obstacle contact report
event/impulse metadata.

## Autoware Docker Path

Use the official Autoware Docker image when the host does not have Autoware message packages installed.

Generate a tiny Autoware fixture:

```bash
scripts/create_autoware_fixture_bag_docker.sh bags/autoware_fixture_success_001
scripts/analyze_autoware_bag_docker.sh \
  bags/autoware_fixture_success_001 \
  outputs/autoware_fixture_success_001
```

Record and analyze a real headless planning simulator run:

```bash
NAV_ANALYZER_RECORD_OVERWRITE=1 \
scripts/record_autoware_planning_run_docker.sh bags/autoware_planning_success_001

scripts/analyze_autoware_bag_docker.sh \
  bags/autoware_planning_success_001 \
  outputs/autoware_planning_success_001
```

## Outputs

NavigationAnalyzer is designed around structured artifacts:

- `navigation_run.json`: canonical normalized run.
- `analysis.json`: metrics, failures, source data, and metadata.
- `diagnosis_pack.json`: compact LLM/CI-oriented run summary with hypotheses, evidence windows, and missing signals.
- `diagnosis.md`: PR-comment-ready Markdown rendering of `diagnosis_pack.json` (verdict, top hypotheses, missing signals).
- `report.md`: human-readable run report.
- `benchmark.json`: comparable multi-run benchmark data.
- `benchmark.md`: human-readable comparison report.

Example failure finding:

```json
{
  "failure_type": "oscillation",
  "timestamp": 123.4,
  "severity": "medium",
  "confidence": 0.78,
  "evidence": {
    "sign_changes": 6
  },
  "possible_causes": [
    "local planner instability",
    "inflation radius too large"
  ]
}
```

Example diagnostic:

```json
{
  "diagnostic_type": "goal_reached_route_progress_mismatch",
  "timestamp": 42.0,
  "level": "warning",
  "confidence": 0.74,
  "summary": "Goal tolerance was satisfied before the matched lanelet route centerline was fully traversed.",
  "evidence": {
    "route_lanelet_progress_ratio": 0.79,
    "route_lanelet_remaining_distance_m": 14.0
  },
  "recommendations": [
    "Check whether the mission goal lies before the end of the final lanelet."
  ]
}
```

## Architecture

The MVP stays intentionally direct:

1. Readers convert ROS2 bags or canonical JSON into `NavigationRun`.
2. Metric functions compute independent `MetricResult` values.
3. Failure rules inspect the same canonical run and produce structured `FailureFinding` records.
4. Reports, benchmark JSON, FastAPI, and the frontend all consume `AnalysisArtifact`.

This avoids premature plugin abstractions while keeping the experiment loop fast.

## Known Limitations

- Nav2 recovery semantics are still shallow; behavior tree status and action feedback parsing are future work.
- Autoware lanelet centerline metrics read lanelet2 OSM left/right boundaries, but corridor polygon distance and regulatory element semantics are still future work.
- Autoware local trajectory is rolling; planner-divergence rules account for trajectory timestamp, but full route-aware evaluation is still needed.
- PointCloud2 clearance depends on `sensor_msgs_py`.
- Plotly currently dominates the frontend bundle size; lazy loading is a later optimization.
- Raw bags are intentionally ignored by git. Commit compact canonical fixtures instead.
